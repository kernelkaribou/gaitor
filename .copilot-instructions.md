# Model gAItor — Copilot Instructions

## Project Overview

Model gAItor is a self-hosted, Docker-based web UI for managing AI model files across a **library** (NAS/source of truth) and one or more **destinations** (inferencing machines). It provides a "smart FTP" experience specifically for AI models — browse, sync, retrieve, rename, and track models with a lightweight, modern interface.

## Architecture

- **Backend**: Python 3.12 + FastAPI — serves REST API, WebSocket for transfer progress, and the built frontend as static files
- **Frontend**: Svelte 5 + Vite + Tailwind CSS — compiles to static JS/CSS, served by FastAPI (no Node.js in production)
- **Metadata**: JSON files stored in the library's `.modelgaitor/` directory. Individual `{uuid}.json` per model for safe writes on network shares, compiled `index.json` for fast reads. No database.
- **Docker**: Multi-stage Dockerfile. `entrypoint.sh` with PUID/PGID support for NAS mount compatibility.

## Key Concepts

- **Library**: The single source of truth for models. Lives on a NAS/network share. Contains the `.modelgaitor/` metadata directory.
- **Destinations**: Recipients of models from the library. Each destination is a Docker volume mount. A sidecar `.{filename}.gaitor.json` file accompanies each synced model for tracking.
- **Categories**: ComfyUI-inspired model type organization (checkpoints, LoRAs, VAE, CLIP, etc.). Customizable by the user.
- **Server-side transfers**: All file copies happen directly between Docker volume mounts. The browser only receives progress updates via WebSocket.

## Development Workflow

### Branching Strategy
- **`main`**: Release branch. Updated via manual merge + tag from `dev`.
- **`dev`**: Integration branch. All feature branches merge here. Pushes auto-build `:dev` Docker tag.
- **Feature branches**: Named `feature/{phase}-{short-description}` (e.g., `feature/p1-docker-setup`).

### Commit Conventions
Use conventional commits: `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`

### Local Development
```bash
# Backend (from repo root)
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
make dev-backend

# Frontend (from frontend/)
npm install
make dev-frontend
```

### Environment
- Python virtual env: `.venv/` (always use, never install globally)
- Node.js: Only for frontend development/build
- Docker: Primary deployment target

## Key Design Decisions

1. **JSON over SQLite**: Network shares (NFS/SMB) have file locking issues with SQLite. Individual JSON files per model minimize write conflicts. An `index.json` cache provides fast reads.
2. **Server-side file transfers**: Models can be 10-50GB+. Transfers happen between Docker volume mounts, not through the browser.
3. **Sidecar metadata on destinations**: Each synced model gets a `.{filename}.gaitor.json` sidecar file. This makes destinations self-describing and tracks rename history.
4. **PUID/PGID in Docker**: Essential for proper file ownership on NAS mounts.
5. **Svelte over vanilla JS**: The app needs real-time progress bars, complex modals, live search, and WebSocket state management. Svelte compiles to vanilla JS with zero runtime overhead.
6. **Port 8487**: "GATR" on a phone keypad.

## API Conventions

- REST endpoints under `/api/`
- WebSocket at `/ws/progress`
- Health check at `/health`
- Version check at `/api/settings/version`
- Frontend served from root `/`
- FastAPI auto-generates OpenAPI docs at `/docs`

## When Making Changes

- Always run `make test` before committing
- Backend changes: ensure Pydantic schemas are updated if API shapes change
- Frontend changes: ensure API client (`frontend/src/lib/api.js`) matches backend endpoints
- Metadata schema changes: consider backward compatibility (existing `.modelgaitor/` directories)
- New dependencies: add to `requirements.in`, then run `pip-compile requirements.in` to update `requirements.txt`
